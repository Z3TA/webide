Running Node.JS scripts

Open any JavaScript (Node.JS) file and press F1 (or click "Run nodejs script" in the menu) to run it ...
Try any of the example files in the /nodejs/ folder.

Unix sockets / named pipes

Use /sock/_somename instead of a port-number. Eg. replace server.listen(80) with server.listen("/sock/_nameOfSocket")

All HTTP (and websocket) request starting with http://%USERNAME%.%DOMAIN%/_underscore will be proxied to your /sock/_underscore unix socket.

You might have to add the unix socket as a prefix/path when using modules such as express and socket.io: Example: express().use("/_unixSocket", express.static('public')); or socketio(server, {path: "/_unixSocket/something"});

If you get Error: EADDRINUSE /sock/_yoursock simply delete the file.
(unix sockets created while running a node.js script are normally deleted by the editor when a node.js script stops)

Relative paths

For example when using the express module to serve static content, the url will be http://%USERNAME%.%DOMAIN%/_sockName
So make sure all paths are relative (not hardcoded with / infront). If you use relative paths you can later change the socket name, for example when putting the app in production.
Example: Linking to /assets/sound/bird.wav from /game/index.htm the url should be ../assets/sound/bird.wav.
Or if index.htm was in the root, the url would be assets/sound/bird.wav (no slash infront)

Deploying to production

Your nodejs scrips will automatically stop when you disconnect from the editor. It's however possible to make the Node.JS script run forever, and have it automatically restart on errors and server reboot.

Press Ctrl + F1 to deploy the currently open Node.js script to "production".
This will copy the entire project into a production enviroment and (re)start the script.
(Make sure you test the script before deploying it.)

  • Ctrl + F1 deployes the script to "production".
  • Shift + Ctrl + F1 restart script "in proction"
  • Ctrl + F3 stop the script "in proction"
  • Shift + Ctrl + F3 remove the script from "proction"

Production log files can be found in the /log/ directory.

Production vs Development enviroment

The "production" enviroment will append prod: true, and myName: %USERNAME% to the process.env (enviroment variables).
While normal (development) will have dev: true (no prod variable).

This can be useful if you for example want to use a different unix socket for production then in development.
